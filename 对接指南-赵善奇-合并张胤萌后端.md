## 0. 给 AI 的硬性规则（必须遵守）

1. **禁止**用张的整棵 `frontend/` 覆盖赵的页面与组件。  
2. **禁止** `git reset --hard` 到别人的分支。  
3. **禁止**为了「更全」去合并张本机未推送的 Phase2，或擅自改远程要求。  
4. 冲突归属见 §2：`frontend` UI 归赵；`backend` 模型/训练/Celery 归张；鉴权/数据归 `main`（杨）。  
5. 前端联调只要求 Phase1 API 能通（见 §4）；没有 `lineage` / 复杂审批页也不算本次失败。  
6. Vite 代理保持 `/api → http://localhost:8000`。

## 2. 合并流程（赵 / AI 逐步执行）

### 2.1 先保存自己的前端

```powershell
cd <Model_Group 仓库根目录>
git status
git checkout -b zhao-frontend    # 已有自己的分支则跳过建分支
git add frontend
git commit -m "chore(frontend): save UI before merging meng phase1"
git fetch origin
```

确认拉到的是 Phase1（有 Celery/M4 骨架即可）：

```powershell
git log --oneline origin/meng -3
# 预期能看到类似：feat(phase1): add Celery, M4/M5 API skeleton...
```

### 2.2 先合杨的 main，再合张的 meng（Phase1）

```powershell
git merge origin/main
# 冲突：frontend UI → 赵；backend 鉴权/数据 → main

git merge origin/meng
# 冲突：按 §1 裁决
```

### 2.3 冲突时（赵的分支上 merge 时：ours=赵，theirs=张）

**页面保留赵的：**

```powershell
git checkout --ours -- frontend/src/views frontend/src/components frontend/src/router
git add frontend/src/views frontend/src/components frontend/src/router
```

**模型 API 封装取张的 Phase1：**

```powershell
git checkout --theirs -- frontend/src/api/model.js
git add frontend/src/api/model.js
```

**张的后端文件取张的：**

```powershell
git checkout --theirs -- backend/app/api/v1/models.py backend/app/api/v1/train.py backend/app/api/v1/infer.py backend/app/api/v1/eval.py
git checkout --theirs -- backend/app/services/normal_model_service.py backend/app/services/normal_eval_service.py
git checkout --theirs -- backend/app/tasks
git add backend/app/api/v1 backend/app/services backend/app/tasks
```

```powershell
git status
git commit -m "merge: meng phase1 backend without overwriting frontend UI"
```

### 2.4 合并后快速检查

```powershell
Select-String -Path frontend\src\api\model.js -Pattern "getMyModels|getModelDetail|getBaselines|submitTrain|getTrainDetail"
Select-String -Path frontend\vite.config.js -Pattern "8000"
```

## 4. Phase1 前端要对的 API

契约可参考仓库内 `docs/api/`（若本地合入后有 v1.0/说明文档以合入后的为准）。  
`request`：`baseURL: '/api'`，Header：`Authorization: Bearer <token>`。

### 4.1 模型 / 训练（`model.js`）

| 函数 | 方法 | 路径 | Phase1 联调 |
|------|------|------|-------------|
| `getMyModels` | GET | `/models` | **必测** |
| `getModelDetail` | GET | `/models/{id}` | **必测** |
| `getBaselines` | GET | `/models/baselines` | **必测** |
| `registerModel` | POST | `/models`（multipart） | 建议 |
| `uploadVersion` | POST | `/models/{id}/versions` | 可选 |
| `submitTrain` | POST | `/train/tasks` | **必测** |
| `getTrainDetail` | GET | `/train/tasks/{id}` | **必测**（可轮询） |
| `stopTrain` | POST | `/train/tasks/{id}/stop` | 可选 |
| `submitInfer` / `getInferResults` | — | `/infer/...` | 有页面再接；Phase1 多为占位结果 |

### 4.2 训练入队（Phase1 注意）

Phase1 骨架里通常有：

- `POST /api/train/tasks` → 创建后多为 `pending_approval`  
- `POST /api/train/tasks/{id}/enqueue` → **联调辅助入队**（投递 Celery）

第一阶段联调可以：

1. 用户提交训练，再调一次 **enqueue** 让任务跑起来，或  
2. 用 admin/有权限账号按张 Phase1 文档操作  

数据集必须是 **`status === 'frozen'`**，否则提交训练会失败。

鉴权 / 数据仍用杨的：`auth.js`、`data.js`、`dataset.js`。

## 5. 怎么起服务 + Phase1 成功标准

### 5.1 后端

```powershell
docker compose up -d
# http://127.0.0.1:8000/docs 应有 Models / Train / Infer / Eval
```

### 5.2 前端

```powershell
cd frontend
npm install
npm run dev
# http://localhost:3000 ，代理到 8000
```

`USE_MOCK` 相关开关保持 **false**（打真实后端）。

### 5.3 Phase1 联调最小成功标准

| 步骤 | 成功 |
|------|------|
| 登录 | Token 可用 |
| 数据列表 | 杨的 `/api/data` 真实数据 |
| 模型列表 / 基线 | `getMyModels` / `getBaselines` 200 |
| 模型详情 | `getModelDetail` 能显示版本等信息 |
| 提交训练 | `submitTrain` 返回 `task_id` |
| 入队 + 查状态 | enqueue（或等价操作）后 `getTrainDetail` 能看到 `queued`→`running`→`completed`（Phase1 多为模拟进度） |

**本次不要求**：版本血缘树页面、管理员审批流页面、真 Docker GPU 训练、真 ONNX 推理。那些是 Phase2/3，张还没推。

## 6. 常见问题

| 现象 | 处理 |
|------|------|
| 没有 `/api/models` | 没合 `origin/meng` 或 api 容器没起 |
| 训练报数据集错误 | 数据集改成 `frozen` |
| 任务一直 `pending_approval` | Phase1 再调 `enqueue`（见 §4.2） |
| 合并后面页没了 | 误用了张的 frontend；从赵分支恢复 `views/components` |
| 有人甩来 Phase2 验收文档 | 可收藏，**本次联调不执行** |
